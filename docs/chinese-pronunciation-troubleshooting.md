# 中文发音排障经验

本文记录 `TextReader` 在接入 `Piper + ONNX Runtime + xiao_ya` 中文模型过程中，围绕“发音奇怪 / 含混 / 单字发虚 / 多音字覆盖不生效 / 与官方样例差异大”等问题的排障过程、关键结论和后续维护建议。

## 1. 问题背景

最初暴露出来的问题并不是单一症状，而是一组彼此叠加的现象：

- 中文朗读整体“怪”，像口齿不清。
- 单字很短时，听起来像“哼了一下”，甚至几乎没声音。
- 导出音频和 `xiao_ya` 官方样例相比，时长、节奏、清晰度差异都很大。
- 即使加载了外部 `g2pW` 生成的拼音 JSON，听感也一度没有明显变化。
- 修改 JSON 里的多音字后，朗读结果仍然不变。

最终证明，这不是一个点状 bug，而是 **文本 → 分句 → 拼音 → symbol → phoneme IDs → ONNX 推理** 整条链路上存在多个细小但关键的不一致。

---

## 2. 这次真正定位出来的关键问题

### 2.1 外部拼音覆盖一开始并没有真正参与句级合成

虽然全文级的 `g2pW` JSON 已经成功加载，但 `PiperTTSEngine` 实际是按句子逐句合成。

早期实现只支持：

- 当前全文文本完全等于 override 里的 `text` 才算命中

这会导致：

- 全文匹配成功
- 但一进入 `SentenceSplitter` 后，每句都不再等于全文
- 最终每句都回退到内置前端

**经验**：如果覆盖/缓存是按全文建的，而下游按句处理，就必须显式做“全文 token → 句级 token”的切分映射，否则功能会表面成功、实际失效。

### 2.2 之前对 Piper 输入 ID 的组装方式，与官方中文实现不一致

在定位过程中，一度参考了通用 Piper 的 phoneme-id 组装方式，并尝试：

- 在每个 phoneme 后加 `PAD`
- 在 `BOS` 后补 `PAD`

这一步虽然让“清晰度”有改善，但仍然不能得到正确的中文读音。

继续核对官方 Python 实现后发现：

- `xiao_ya` 这类 `phoneme_type: "pinyin"` 的中文模型，**并不完全走通用路径**
- 官方存在单独的中文前端实现
- 中文模型的 `PAD` 插入规则是 **按一个完整拼音组结束后补 `PAD`**，而不是每个 symbol 后都补

这是本次修复发音问题的核心转折点之一。

### 2.3 中文模型的前端逻辑是专用的，不能拿通用语言规则直接套

这次最大的一个教训是：

> 不要默认中文模型和通用多语言 phoneme 流水线只差一个“词典”。

对于 `xiao_ya` 这类 `phoneme_type: "pinyin"` 模型，至少有这些点必须和官方中文路径对齐：

- 拼音 syllable 的拆分方式
- 零声母表示
- 声调 token 的边界含义
- `PAD` 的插入时机
- 标点/空格在输入中的结构作用

只要这里有细小偏差，中文发音就会整体失真。

### 2.4 `g2pW` 是否“用上了”，必须用数据证明，不能靠听感猜

一开始虽然已经支持加载外部拼音 JSON，但如果只凭耳朵听，很难判断到底是：

- `g2pW` 没生效
- `g2pW` 生效了，但后续 symbol 映射不对
- 还是推理输入格式仍然不对

后来通过导出调试 JSON，增加这些字段后，问题就容易定位得多：

- `usedExternalOverride`
- `tokens`
- `symbols`
- `phonemeIDs`
- 句级切分结果
- 当前使用的拼音模式

**经验**：TTS 问题一定要导出中间层，不要只听最终音频猜原因。

### 2.5 文本规范化差异会让外部覆盖静默失效

即使 `g2pW` JSON 和 App 内加载的文本来自同一个源文件，仍然可能不完全相等，常见原因包括：

- UTF-8 BOM
- `CRLF` / `LF` 差异
- 首尾空白
- 不同读取方式对换行的处理不同

后来统一了：

- Python 生成端的文本规范化
- Swift 文件导入时的 BOM / 换行规范化
- 匹配逻辑中的规范化

并增加了 mismatch 诊断和高亮显示，才让这条链路可维护。

**经验**：任何依赖“字符串完全相等”的功能，都必须先统一规范化策略。

## 3. 这次哪些做法有效，哪些是弯路

### 有效做法

#### 3.1 用官方样例做 A/B 对比

把 `sample.txt` 和 `speaker_0.mp3` 当作固定基准非常有用，可以快速判断：

- 当前偏差是节奏问题还是发音体系问题
- 改动是否真的朝正确方向收敛

#### 3.2 给导出音频配套导出调试 JSON

这是整个排障过程中最有价值的工具之一。它让每次实验都可以回答：

- 这次到底用了哪套 token
- 外部拼音覆盖是否命中
- 最终送进模型的 IDs 是什么

#### 3.3 逐步做 A/B 实验，而不是一次改很多假设

比如这次曾经专门测试：

- `py-ortho`
- `py-zero`

虽然最后证明 `y/w` 零声母归一不是主因，但这个实验很有价值，因为它明确排除了一个方向。

#### 3.4 直接去看官方 Python 运行时源码，而不是继续猜

真正解决问题的关键节点，不是继续凭经验微调，而是去看：

- 官方 Python `piper` 的中文实现
- 官方中文模型到底如何把拼音转成 IDs

**经验**：第三方模型/运行时问题，优先对齐上游实现，不要先自行脑补“差不多的规则”。

### 弯路 / 次优路径

#### 3.5 只调 `length_scale` 不能解决字音问题

`length_scale` 的确影响了与官方样例的时长接近程度，但只能修复：

- 节奏
- 句子整体快慢

它不能修正：

- 错音
- 中文边界不对
- 外部覆盖未生效
- ID 组装错误

#### 3.6 只靠“听起来像采样率不对”这种直觉容易走偏

最开始“声音怪”很容易让人怀疑：

- 采样率不对
- 播放引擎有问题
- `TimePitch` 引入失真

这些方向并非完全没价值，但事实证明真正的主问题是在中文前端和模型输入规则，而不是纯播放层。

#### 3.7 单独修多音字词典，不足以解决系统性错音

多音字覆盖可以修一部分局部问题，但如果整个 `pinyin -> ids` 逻辑本身和官方路径不一致，那么就会出现：

- 清晰度上来一点
- 但整体字音体系仍然都不对

## 4. 最终沉淀下来的维护原则

### 4.1 先观测，再判断

优先保留并使用这些诊断信息：

- 原始文本
- 分句结果
- 句级 tokens
- symbols
- phoneme IDs
- 是否命中外部覆盖

### 4.2 中文路径要单独对待

只要看到这些特征，就不要假设可以直接复用通用 phoneme 流水线：

- `phoneme_type == "pinyin"`
- 中文 `g2pW` / `g2p` 前端
- 声调数字直接进入模型输入
- 中文专用 `phonemize` 实现

### 4.3 全文处理与分句处理之间必须做映射

外部覆盖、缓存、诊断、导出等功能，如果有的按全文、有的按句子，就必须明确维护：

- token 索引映射
- 标点是否占位
- 句子切分后的 token slice

### 4.4 任何精确匹配前都要做规范化

至少统一：

- BOM
- 换行符
- 首尾空白
- 编码读取方式

### 4.5 调试工具要长期保留

这次补上的这些能力，不应视作临时排障代码，而应视为长期可维护性的基础设施：

- 导出调试 JSON
- mismatch 高亮
- mismatch 摘要
- override 命中状态提示
- 导出文件名带关键实验参数

## 5. 对后续开发的具体建议

### 5.1 保留并继续完善导出 JSON

建议继续保留以下字段，并在未来新增更多必要信息：

- `usedExternalOverride`
- `tokens`
- `symbols`
- `phonemeIDs`
- `pinyinMode`
- 当前 `length_scale`
- 当前句的原文与模型输入之间的映射关系

### 5.2 如果未来切换中文模型，优先查它的官方前端

不要复用当前对 `xiao_ya` 成功的全部假设。新模型应优先确认：

- 它是否仍然使用 `phoneme_type: "pinyin"`
- 是否有新的 symbol 规则
- 是否仍依赖 `g2pW`
- 是否有专用中文推理路径

### 5.3 优先做“更完整中文 phonemizer”，但要以可验证方式推进

如果后续要继续提升中文质量，建议沿这条顺序推进：

1. 保持与官方中文路径一致
2. 对难词、数字、日期、英文混读做增量增强
3. 每次改动都保留 A/B 对照和可导出的中间层

### 5.4 对“单字/极短输入”保留特殊保护

这次还暴露了中文模型在极短 utterance 上更容易出现：

- 发音太弱
- 起音被吃掉
- 听感像“哼一下”

后续维护时，应继续关注：

- 极短输入的边界稳定化
- 超短音频的淡入淡出策略
- 是否需要继续保留轻度归一化与弱静音裁剪

---

## 6. 一句话总结

这次排障最大的经验是：

> 中文 TTS 的“发音奇怪”不能只靠听感调参，必须把 **文本 → token → symbol → IDs → 模型输入** 整条链路做成可观测、可对齐、可验证的系统；尤其对 `phoneme_type: "pinyin"` 的中文模型，必须优先对齐官方中文前端实现，否则哪怕只有细小偏差，也会导致整体发音严重失真。
